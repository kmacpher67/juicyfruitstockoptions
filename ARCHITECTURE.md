# Architecture Overview

This document describes the high-level architecture of the Juicy Fruit Stock Options trading platform.

## High-Level Hierarchy

The application follows a Service-Oriented Architecture (SOA) style, structured in layers:

1.  **Frontend**: React-based UI (in `frontend/`) interacting with the API.
2.  **API Layer**: FastAPI (`app/api/routes.py`) exposing REST endpoints.
3.  **Service Layer**: Encapsulates business logic (`app/services/`).
4.  **Data Layer**: MongoDB (`app/database.py` and direct driver usage).
5.  **Scheduler**: APScheduler (`app/scheduler/`) for background jobs.

## Core Services

### 1. Roll Service (`app/services/roll_service.py`)
*   **Responsibility**: Analyzes options positions to find optimal "rolling" opportunities (extending duration, adjusting strike).
*   **Key Features**: Smart Roll logic, Greeks calculation, Signal integration.
*   **Dependencies**: `SignalService`, `OpportunityService`.

### 2. Dividend Service (`app/services/dividend_scanner.py`)
*   **Responsibility**: Scans for Dividend Capture opportunities and manages Corporate Events.
*   **Key Features**:
    *   **Dividend Scanner**: Identifies high-yield buy-write opportunities before ex-dividend dates.
    *   **Calendar Generator**: Generates ICS files for corporate events (Earnings, Ex-Div).
*   **Dependencies**: `RollService`, `SignalService`, `OpportunityService`, `NewsService`.

### 3. Jobs / Scheduler (`app/scheduler/jobs.py`)
*   **Responsibility**: Orchestrates background tasks.
*   **Key Jobs**:
    *   `run_dividend_scan_wrapper`: Triggers Dividend Scanner.
    *   `run_stock_live_comparison`: Validates live prices vs recorded state.
    *   `run_ibkr_sync`: Syncs portfolio data from IBKR.
    *   `run_recommendation_scans`: Generates and persists trading signals.

### Scanner Services Hierarchy

Scanners are organized by trading phase, each targeting a distinct segment of the wheel/options strategy lifecycle:

**Phase 1 — Entry Scanner: `CSP_Optimizer`** *(Proposed)*
*   **Goal**: Find high-yield Cash Secured Puts (CSPs) on quality underlyings.
*   **Focus**: Strike selection below support, IV rank filtering, yield-to-expiry ranking.
*   **Output**: CSP opportunities ranked by Annualized Yield %.

**Phase 2 — Yield Scanner: `OptionsAnalyzer`** *(Current — `app/services/options_analysis.py`)*
*   **Goal**: Maximize ARIF % (Annualized Return on Invested capital from premium).
*   **Focus**: Covered call selection across held positions; strike, DTE, and IV optimization.
*   **Output**: Covered call opportunities with yield and premium metrics.

**Phase 3 — Tactical Scanners**

| Scanner | File | Status | Purpose |
|---|---|---|---|
| `ExpirationScanner` | `expiration_scanner.py` | Active | Identifies positions with DTE < 7 for rolling or closing decisions |
| `DividendScanner` | `dividend_scanner.py` | Active | Scans for ex-div dates and buy-write capture opportunities |
| `MomentumScanner` | *(Proposed)* | Proposed | Scans for Juicy Buy Calls by correlating technical breakouts with low IV |

## Data Flow & State Management

### Single Source of Truth
**MongoDB** (`stock_analysis` database) is the Single Source of Truth for all persistent state.
*   **Portfolio State**: `ibkr_holdings` collection (Synced from IBKR).
*   **Opportunities**: `opportunities` collection (Generated by Scanners).
*   **Corporate Events**: `corporate_events` collection (Aggregated from Data Providers).
*   **Config**: `system_config` collection.

### Scanners to Collections Flow

1.  **Scanners** (Dividend, Expiration, Signal) are triggered by **Jobs**.
2.  **Scanners** fetch raw data from external providers (IBKR, Yahoo Finance).
3.  **Scanners** process data using `SignalService` (AI/Algo models).
4.  **Results** are persisted via `OpportunityService` to the **`opportunities`** collection.
5.  **Corporate Events** (specific to Dividend Scanner) are persisted to the **`corporate_events`** collection.
6.  **Frontend** reads from these collections via API endpoints.

## New Collection Interaction
The newly added **`corporate_events`** collection interacts as follows:
*   **Writer**: `DividendScanner.generate_corporate_events_calendar()` fetches data and upserts documents into `corporate_events`.
*   **Reader**: The same method reads from `corporate_events` to generate the `.ics` file.
*   **Context**: It uses `ibkr_holdings` to determine which tickers to track.
